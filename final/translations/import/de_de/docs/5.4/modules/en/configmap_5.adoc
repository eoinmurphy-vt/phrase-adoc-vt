== Kubernetes ConfigMap

{product-name} unterstützt die automatisierte Konfiguration mit der Kubernetes ConfigMap-Funktion. Dies ermöglicht die Bereitstellung von {product-name} Containern mit den entsprechenden Konfigurationen, Integrationen und anderen Einstellungen auf automatisierte Weise.

Die Einstellung 'always_reload: true' kann in jede ConfigMap-YAML hinzugefügt werden, um das Neuladen dieser YAML jedes Mal zu erzwingen, wenn der Controller startet (Version 4.3.2+). Andernfalls wird die ConfigMap nur beim ersten Start oder nach einem vollständigen Clusterneustart geladen (siehe Abschnitt über persistente Speicherung unten).

=== Vollständige Beispiel {product-name} ConfigMap (initcfg.yaml)

Die neueste ConfigMap finden Sie https://raw.githubusercontent.com/neuvector/manifests/main/kubernetes/5.4.0/initcfg.yaml[hier].

Das Beispiel wird auch unten angezeigt. Dies enthält alle verfügbaren Einstellungen. Bitte entfernen Sie die nicht benötigten Abschnitte und bearbeiten Sie die benötigten Abschnitte. 

[NOTE]
====
Wenn Sie configmap in einem Geheimnis verwenden, siehe Abschnitt unten für Formatänderungen.

=== Beispiele für föderierte ConfigMap (fedinitcfg.yaml)

NeuVector v5.4.0 unterstützt die Automatisierung der Föderation über die ConfigMap. Unten sind Beispiel `fedinitcfg.yaml` Konfigurationen aufgeführt, die je nach Anwendungsfall auf Ihre primären und verwalteten Cluster angewendet werden können.

Beispiel `fedinitcfg.yaml` für den primären Cluster:

[,yaml]
----
# Optional. If specified, it overwrites the cluster name specified in system configuration
Cluster_Name: cluster-primary-43
# Required and must be the same on primary cluster and managed clusters
# It must be 36 characters long, i.e., 32 hex characters grouped as 8-4-4-4-12 and separated by four hyphens
Join_Token: 2be93d8f-d42a-44fc-9d33-7c1a6153066b
# Required: REST server/port of the neuvector-svc-controller-fed-master service
Primary_Rest_Info:
    Server: 10.1.10.43
    Port: 30020
# Optional. Supported value: https
Use_Proxy: ""
# Optional. Whether federal repo scan data deployment is enabled (for primary cluster only)
Deploy_Repo_Scan_Data: false
----

Beispiel `fedinitcfg.yaml` für verwaltete Cluster:

[,yaml]
----
# Optional. If specified, it overwrites the cluster name specified in system configuration
Cluster_Name: cluster-managed-42
# Required and must be the same on primary cluster and managed clusters
# It must be 36 characters long, i.e., 32 hex characters grouped as 8-4-4-4-12 and separated by four hyphens
Join_Token: 2be93d8f-d42a-44fc-9d33-7c1a6153066b
# Required: REST server/port of the neuvector-svc-controller-fed-master service
Primary_Rest_Info:
    Server: 10.1.10.43
    Port: 30020
# Optional, for managed cluster only. REST server/port of the neuvector-svc-controller-api service
Managed_Rest_Info:
    Server: 10.1.10.42
    Port: 30010
# Optional. Supported value: https
Use_Proxy: ""
----

== Schützen Sie sensible Daten mit einem Geheimnis.

Wenn sensible Daten in einigen Abschnitten des Configmaps enthalten sein sollen, kann ein Geheimnis für diese Abschnitte mit sensiblen Daten erstellt werden.

Erstellen Sie beispielsweise das ConfigMap für NICHT-sensible Abschnitte wie passwordProfile und role:

[,shell]
----
kubectl create configmap neuvector-init --from-file=$HOME/init/passwordprofileinitcfg.yaml --from-file=$HOME/init/roleinitcfg.yaml -n neuvector
----

Erstellen Sie dann ein Geheimnis für Abschnitte mit sensiblen Daten, wie zum Beispiel:

[,shell]
----
kubectl create secret generic neuvector-init --from-file=$HOME/init/eulainitcfg.yaml --from-file=$HOME/init/ldapinitcfg.yaml --from-file=$HOME/init/oidcinitcfg.yaml --from-file=$HOME/init/samlinitcfg.yaml --from-file=$HOME/init/sysinitcfg.yaml --from-file=$HOME/init/userinitcfg.yaml -n neuvector
----

[IMPORTANT]
====
Entfernen Sie das Pipe-Zeichen '|' in jedem Abschnitt, wie unten gezeigt.
====

Beachten Sie die Entfernung des Pipe-Zeichens unten, wenn Sie Configmap-Abschnitte in einem Geheimnis verwenden, aktiviert auf true, und kommentieren Sie den Abschnitt aus, der im Geheimnis enthalten sein soll.

[,yaml]
----
secret:
    # NOTE: files defined here have preferrence over the ones defined in the configmap section
    enabled: true
    data:
      eulainitcfg.yaml:
        license_key: 0Bca63Iy2FiXGqjk...
      #   ...
      # ldapinitcfg.yaml:
      #   directory: OpenLDAP
      #   ...
      # oidcinitcfg.yaml:
      #   Issuer: https://...
      #   ...
      # samlinitcfg.yaml:
      #   ...
      # sysinitcfg.yaml:
      #   ...
      # userinitcfg.yaml:
      #   ...
----

Nachdem der Controller bereitgestellt wurde, werden alle Konfigurationsdateien sowohl aus dem Configmap als auch aus dem Geheimnis im Ordner /etc/config gespeichert.

Beachten Sie, dass das Geheimnis in den Standard-Kubernetes- und OpenShift-Controller xref:kubernetes.adoc#_deploy_suse_security[Deployment-YAML-Dateien] unter Volumes referenziert wird.

== ConfigMaps und Persistenter Speicher

Sowohl die ConfigMaps als auch das xref:production.adoc#_backups_and_persistent_data[Backup des persistenten Speichers] werden nur gelesen, wenn ein neuer {product-name} Cluster bereitgestellt wird oder der Cluster ausfällt und neu gestartet wird. Sie werden während der Rolling-Upgrades nicht verwendet.

Das Backup der Konfiguration des persistenten Speichers wird zuerst gelesen, dann werden die ConfigMaps angewendet, sodass die Einstellungen des ConfigMaps Vorrang haben. Alle ConfigMap-Einstellungen (z. B. Updates) werden ebenfalls im persistenten Speicher gespeichert.
