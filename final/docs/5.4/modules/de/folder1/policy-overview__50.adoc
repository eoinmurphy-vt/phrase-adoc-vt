== {product-name} Sicherheitspolitik

Um Verstöße gegen das normale Verhalten zu erkennen, unterhält {product-name} eine Sicherheitsrichtlinie, die über die GUI, CLI, CRD oder REST API verwaltet werden kann.

=== Gruppen

Dies bietet die primäre Ansicht von Dienstgruppen und benutzerdefinierten Gruppen, um den Modus (Erkennen, Überwachen, Schützen) für jeden Dienst einzustellen und Regeln zu verwalten. Gruppen werden automatisch von {product-name} erstellt, aber es können auch eigene Gruppen hinzugefügt werden. Die Regeln für jede Gruppe werden automatisch von {product-name} erstellt, wenn die Container in Betrieb genommen werden. Container-Gruppen können einen geteilten Richtlinienmodus haben, bei dem die Prozess-/Dateiregeln einen anderen Durchsetzungsmodus haben als die Netzwerkregeln, wie xref:modes.adoc#_network_service_policy_mode[hier] beschrieben.

Um eine Gruppe zur Ansicht oder Verwaltung auszuwählen, aktivieren Sie das Kontrollkästchen neben der Gruppe. Hier werden Prozessprofilregeln, Dateizugriffsregeln, DLP und benutzerdefinierte Compliance-Prüfungen verwaltet. Netzwerkregeln können hier eingesehen werden, werden aber in einem separaten Menü verwaltet.
Netzwerk- und Antwortregeln in {product-name} werden über die Felder '`from`' und '`to`' erstellt, die eine Gruppe als Eingabe erfordern. Eine Gruppe kann eine Anwendung sein, aus Bildbezeichnungen, DNS-Namen oder anderen benutzerdefinierten Gruppierungen abgeleitet werden. DNS-Subdomänen werden unterstützt, z. B. *.foo.com. Es können auch IP-Adressen oder Subnetze verwendet werden, was für die Kontrolle des Ein- und Austritts von nicht containerisierten Workloads nützlich ist.

image:groups_node_demo.png[Gruppen]

Zu den reservierten Gruppennamen, die automatisch von {product-name} erstellt werden, gehören:

* Behältnisse. Alle laufenden Container.
* Extern. Verbindungen, die in den Cluster eintreten (Ingress).
* Knotenpunkte. Knoten oder Hosts, die durch {product-name} identifiziert werden.

Im Menü "Gruppen" kann auch die Funktion "Gruppenrichtlinie exportieren" ausgeführt werden. Dadurch wird die Sicherheitsrichtlinie (Regeln) für die ausgewählten Gruppen als yaml-Datei im Format der benutzerdefinierten Ressourcendefinition (CRD) {product-name} exportiert, die überprüft und dann in anderen Clustern eingesetzt werden kann.

Beachten Sie, dass der Status der Container einer Gruppe in der Richtlinie -> Groups -> Members angezeigt wird, die den {product-name} Schutzmodus (Discover, Monitor, Protect) angibt. Wenn der Container im Zustand "Beenden" angezeigt wird, befindet er sich noch auf dem Host, ist aber angehalten. Durch das Entfernen des Containers wird dieser aus dem Exit-Status entfernt.

=== Netzregeln

Eine Liste von Whitelist- und Blacklist-Regeln, die {product-name} durchsetzen kann. {product-name} kann im Erkennungsmodus eine Reihe von Whitelist-Regeln automatisch erkennen und erstellen. Regeln können bei Bedarf manuell hinzugefügt werden.

{product-name} erstellt im Erkennungsmodus automatisch Whitelist-Regeln für die Schicht 7 (Anwendungsschicht), indem es die Netzwerkverbindungen beobachtet und Regeln zur Durchsetzung von Anwendungsprotokollen erstellt.

{product-name} verfügt außerdem über eine integrierte Erkennung von Netzwerkangriffen, die unabhängig vom Modus (Entdecken, Überwachen, Schützen) immer aktiviert ist. Zu den festgestellten Netzwerkbedrohungen gehören DDoS-Angriffe, Tunneling und SQL-Injection. Eine vollständige Liste der integrierten Bedrohungserkennung finden Sie im Abschnitt Netzwerkregeln.

DLP-Regeln (Data Loss Prevention) können auch auf Container-Gruppen angewandt werden, um die Netzwerk-Nutzlast auf potenziellen Datendiebstahl oder Datenschutzverletzungen wie unverschlüsselte Kreditkartendaten zu untersuchen. Verstöße können blockiert werden. Weitere Informationen zum Erstellen und Anwenden von DLP-Filtern finden Sie im Abschnitt über DLP.

=== Prozessprofil und Dateizugriffsregeln

{product-name} verfügt über eine integrierte Erkennung verdächtiger Prozesse und Dateiaktivitäten sowie über eine Baselining-Technologie für Container. Die integrierte Erkennung umfasst Prozesse wie Port-Scanning (z. B. NMAP), Reverse Shell und sogar Privilegienerweiterungen zu Root. Systemdateien und Verzeichnisse werden automatisch überwacht. Jeder Dienst, der von {product-name} entdeckt wird, erstellt eine Baseline des Prozess- und Dateiverhaltens von '`normal`' für diesen Containerdienst. Diese Regeln können bei Bedarf angepasst werden.

=== Antwortregeln

Reaktionsregeln ermöglichen es Benutzern, Aktionen zu definieren, um auf Sicherheitsereignisse zu reagieren. Zu den Ereignissen gehören Bedrohungen, Verstöße, Vorfälle und Ergebnisse von Schwachstellen-Scans. Zu den Aktionen gehören Container-Quarantäne, Webhooks und Unterdrückung von Alarmen.

Reaktionsregeln bieten eine flexible, anpassbare Regel-Engine zur Automatisierung von Reaktionen auf wichtige Sicherheitsereignisse.

=== Regeln für die Zulassungskontrolle

Die Regeln der Zulassungskontrolle erlauben oder blockieren Einsätze. Weitere Einzelheiten finden Sie in diesem Abschnitt unter Zulassungskontrollen.

=== DLP- und WAF-Sensoren

Data Loss Prevention (Schutz vor Datenlecks) und WAF-Regeln können für jede ausgewählte Containergruppe aktiviert werden. Dabei wird Deep Packet Inspection verwendet, um den auf regulären Ausdrücken basierenden Abgleich auf die Netzwerk-Nutzdaten anzuwenden, die in die ausgewählte Containergruppe eingehen oder sie verlassen. Eingebaute Sensoren für Kreditkarten und US-Sozialversicherungsnummern sind als Beispiele enthalten, und es können benutzerdefinierte reguläre Ausdrücke hinzugefügt werden.

=== Migration, Sicherung, Import/Export

Die Migration der Sicherheitsrichtlinie kann über CRD, REST API oder Import/Export erfolgen. Beispielsweise können gelernte und benutzerdefinierte Regeln in einer Staging-Umgebung eine oder mehrere CRD-Yaml-Dateien für die Bereitstellung in der Produktionsumgebung erzeugen.

Die Sicherheitsrichtlinie für {product-name} kann unter Einstellungen -> Konfiguration exportiert und importiert werden. Es wird empfohlen, vor einem Update von {product-name} auf eine neue Version eine Sicherungskopie der gesamten Konfiguration zu erstellen.

[IMPORTANT]
====
Wenn Sie ALLES importieren (Config und Policy), wird alles überschrieben, einschließlich der Hauptanmeldedaten für den Administrator. Vergewissern Sie sich vor dem Importieren, dass Sie den Hauptadministrator-Login für die importierte Datei kennen.
====
