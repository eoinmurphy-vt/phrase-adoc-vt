== Sensible Informationen in den Containerprotokollen von NeuVector Manager

*CVE-ID:* CVE-2025-46808*CVSS-Score:* 6.8 - https://nvd.nist.gov/vuln-metrics/cvss/v3-calculator?vector=AV:N/AC:L/PR:L/UI:R/S:C/C:H/I:N/A:N&version=3.1[AV:N/AC:L/PR:L/UI:R/S:C/C:H/I:N/A:N]*CWE:* https://cwe.mitre.org/data/definitions/532[: Einfügung von sensiblen Informationen in die Protokolldatei] CWE-532

=== Betroffene Versionen

* Alle Versionen früher als `+5.0.0+`
* Versionen von `+5.0.0+` bis zu und einschließlich `+5.4.4+`

*Festgelegte Version: `+5.4.5+`*

=== Auswirkungen

In den NeuVector-Versionen bis einschließlich `+5.4.4+` wurde eine Schwachstelle entdeckt, durch die sensible Informationen in das Protokoll des Manager-Containers gelangen können. Die folgenden Felder können offengelegt werden:

[cols="1,2,2,2,2", options="header"]
|===
|Feld |Feld Beschreibung |Wo es auftaucht |Vervielfältigung |Umwelt

|`+X-R-Sess+`
|Ranchers Sitzungs-Token für Single Sign-On
|Kopfzeile der Anfrage
|Melden Sie sich über die Rancher UI an und greifen Sie auf NeuVector SSO zu
|Rancher mit NeuVector SSO

|`+personal_access_token+`
|GitHub / Azure DevOps-Token
|Körper der Anfrage
|Übermitteln Sie die Konfiguration des entfernten Repositorys unter *Konfiguration > Einstellungen*
|NeuVector

|`+token1.token+`
|Sitzungs-Token des NeuVector-Benutzers
|Antwortkörper
|GET-Anfrage an `+https://<neuvector-ui-url>/user?name=<username>+`
|NeuVector

|`+rekor_public_key+`, `+root_cert+`, `+sct_public_key+`
|Öffentlicher Rekor-Schlüssel, Root-Zertifikat, öffentlicher SCT-Schlüssel in privater Vertrauensbasis
|Körper der Anfrage
|Erstellen/Aktualisieren der privaten Vertrauensbasis auf der Sigstore-Seite
|NeuVector

|`+public_key+`
|Öffentlicher Schlüssel des Verifizierers
|Körper der Anfrage
|Prüfer auf der Sigstore-Seite erstellen/aktualisieren
|NeuVector
|===

[NOTE]
====
NeuVector-Installationen, die weder Rancher SSO noch Remote Repository Configuration verwenden, sind von diesem Problem *nicht betroffen*.
====

In der gepatchten Version ist der Wert `+X-R-Sess+` aus Sicherheitsgründen teilweise maskiert. Andere Felder wie `+personal_access_token+`, `+token+`, `+rekor_public_key+`, `+root_cert+`, `+sct_public_key+` und `+public_key+` werden nicht mehr protokolliert, da die Protokollierung des Anfragekörpers entfernt wurde.

[NOTE]
====
* Der Schweregrad dieser Sicherheitslücke hängt von Ihrer Protokollierungsstrategie ab:
  ** *Lokale Protokollierung (Standard):* Begrenzt die Stoßbelastung.
  ** *Externe Protokollierung:* Erhöht das Risiko, abhängig von der Sicherheitslage Ihres externen Protokollsammlers.
* Der Schweregrad der Auswirkungen kann auch von den Berechtigungen der offengelegten Anmeldeinformationen abhängen.

Siehe auch: https://attack.mitre.org/techniques/T1552/[MITRE-Technik T1552: Ungesicherte Anmeldeinformationen].
====

=== Aufnäher

Gepatchte Versionen sind `+5.4.5+` und höher. Benutzer sollten nach dem Upgrade alle GitHub-Token, die in der Remote-Repository-Konfiguration verwendet werden, rotieren.

=== Workarounds

Derzeit sind keine Umgehungsmöglichkeiten verfügbar. Aktualisieren Sie so bald wie möglich auf die korrigierte Version.
